You are an elite senior systems architect, security engineer, backend engineer, DevOps engineer, protocol engineer, and product designer. Your task is to design and build a production-ready, self-hostable, decentralized Fediverse server called CLIverse.

CLIverse is a custom ActivityPub-powered Fediverse instance with no web UI at all. The entire user experience must happen through SSH and an interactive CLI shell-like environment. Users connect to the instance like this:

ssh username@post.rbxl.eu -p 6969

Administrators connect like this:

ssh admin@post.rbxl.eu -p 6969

The project must run on Linux servers, be fully deployable with Docker Compose, work cleanly behind Caddy, and be built for real-world hosting, federation, scaling, and security. The platform must be truly decentralized and really use ActivityPub, not a fake local-only Twitter clone.

The system must be designed as a serious, secure, polished, production-grade project. It must feel like a real SSH-native social network client/server environment.

Core concept

Build a Fediverse server where:
	•	there is no browser interface
	•	users interact only over SSH
	•	the SSH session launches them into a custom interactive CLI app
	•	every social action is done via commands
	•	federation is handled via ActivityPub
	•	admins manage the instance either through a dedicated secure admin SSH flow or dedicated admin commands
	•	the platform supports user accounts, posts, follows, replies, boosts, profiles, remote actors, federation controls, moderation, logs, and administration

This must not behave like a normal Linux shell. After SSH authentication, users should enter a restricted, purpose-built CLIverse shell with a professional TUI/CLI-like experience.

	•	posts must be federated, remote users must be followable, and instance federation controls must affect actual federation logic

	•	SSH authentication, account lifecycle, rate limits, brute-force protections, secure password handling, audit logging, sandboxing, role separation, secrets management, container security, admin isolation, federation abuse controls, and denial-of-service mitigations must all be covered

Required account creation flow

The system must support a secure account onboarding process with this exact concept:
	•	during account creation, the user must provide an administrator with their own SSH public key
	•	the admin creates the account for them
	•	the system or admin assigns a randomly generated initial password
	•	the user logs in using their SSH flow and then must change the initial password with a command:
settings update_password
	•	this command must ask for:
	•	old password
	•	new password
	•	new password again for confirmation
	•	the initial password must be treated as temporary and rotation should be enforced or strongly encouraged
	•	SSH key registration and password-based confirmation logic must be designed carefully and securely

Clarify and design whether normal users authenticate with:
	•	SSH key only
	•	SSH key + app password
	•	password only for first login then key binding
	•	keyboard-interactive challenge inside the CLIverse shell
	•	or a layered hybrid method

Choose the architecture that is most secure and practical, but preserve the required user onboarding concept.

Required admin access flow

There must be an administrator management path using SSH.

Requirements:

	•	it must require a shared admin-only SSH key setup
	•	after key validation, the system must also require a really secure, unchangeable password
	•	the purpose of the extra password challenge is to confirm the private/public SSH material was not shared outside the admin group
	•	this admin environment must be isolated from normal user mode
	•	admin access must include elevated commands for moderation, federation management, user creation, logs, health, tasks, and safety controls

You should also evaluate whether a shared admin account is wise from a security and auditing perspective. If it is insecure, then preserve the requested behavior while improving it with:

	•	tamper-evident logs

Design the best production-safe version while keeping the requested SSH admin concept intact.
