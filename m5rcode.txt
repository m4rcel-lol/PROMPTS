## SECTION 1 — LANGUAGE IDENTITY & PHILOSOPHY

### 1.1 Name & Branding

- Language name: **m5rcode** (always lowercase, never stylized differently)
- File extension: `.m5r`
- Bytecode cache extension: `.m5rc`
- REPL prompt: `m5r>> `
- Version string format: `m5rcode-[MAJOR]x[MINOR]z[PATCH]` (e.g. `m5rcode-1x0z0`)
- Error prefix: `[M5R-FAULT]`
- Warning prefix: `[M5R-WARN]`

### 1.2 Core Philosophy (these are LAWS of the language)

1. **Clarity is earned, not given.** The language offers zero hand-holding. Error messages tell you what broke and nothing else.
1. **Everything is a symbol.** There are no English keywords. Every construct uses operator-like symbols or cryptic sigils.
1. **State is explicit.** Variables do not exist. Only registers, stacks, and named memory cells do.
1. **Execution flows backward by default.** The main block executes its statements from bottom to top unless a forward-flow sigil is used.
1. **Types are masks.** All values are raw 64-bit words. The type is a mask applied at operation time, not stored with the value.
1. **Functions do not return.** They deposit their result into a designated output cell and vanish.
1. **The programmer is always wrong.** The compiler assumes every program has a bug until proven otherwise, running a mandatory pre-flight lint pass.

## SECTION 2 — TOOLCHAIN ARCHITECTURE

Build the following executables from scratch in C:

### 2.1 `m5rc` — The Compiler

- Reads `.m5r` source files
- Outputs `.m5rc` bytecode files (custom binary format, defined in Section 7)
- Passes: Lexer → Parser → Semantic Analyzer → IR Generator → Optimizer → Bytecode Emitter
- Flags:
  - `--strict` : Enables paranoid mode (any ambiguity = fatal error)
  - `--emit-ir` : Dumps the Intermediate Representation to stdout
  - `--no-opt` : Skips the optimizer pass
  - `--trace` : Annotates bytecode with source line numbers
  - `--wall` : Enables all warnings

### 2.2 `m5rv` — The Virtual Machine

- Executes `.m5rc` bytecode
- Stack-based VM with a register file of exactly **16 named registers** (`R0`–`R15`)
- Has its own heap allocator (implement a simple slab allocator)
- Flags:
  - `--memlimit <bytes>` : Hard memory cap
  - `--trace` : Print each instruction as it executes
  - `--sandbox` : Disable all I/O syscalls

### 2.3 `m5r` — The Runner (wrapper)

- Accepts a `.m5r` file, compiles it in-memory, and immediately executes it
- Behaves like `python script.py` but for m5rcode
- Usage: `m5r run <file.m5r>`

### 2.4 `m5rsh` — The REPL

- Interactive shell
- Displays `m5r>> ` prompt
- Supports multiline input with `...` continuation prompt
- Stores session history in `~/.m5r_history`
- Command `:dump` prints current VM state
- Command `:reset` clears all memory cells
- Command `:load <file>` loads a `.m5r` file into session

### 2.5 `m5rpkg` — The Package Manager

- Packages are stored as `.m5rp` archives (tar.gz with a manifest)
- Manifest file: `pkg.m5rmf` (custom format, defined below)
- Commands:
  - `m5rpkg install <name>`
  - `m5rpkg remove <name>`
  - `m5rpkg publish <dir>`
  - `m5rpkg list`
- Registry format: a flat JSON file served from a configurable URL

## SECTION 3 — SYNTAX SPECIFICATION (THE WEIRD PART)

This is where m5rcode becomes truly alien. Study every rule.

### 3.1 Source File Structure

Every `.m5r` file MUST begin with a **file sigil block**:

```
%%m5r
@ident: <module_name>
@ver: <semver_string>
@author: <string>
%%
```

Anything before `%%m5r` is treated as a file-level comment and ignored.
Anything after the closing `%%` is live code.

### 3.2 Comments

```
~~ this is a single line comment
~~[ this is a
    multiline comment ]~~
```

### 3.3 Memory Cells (Variables)

There are **no variables**. There are **cells**. Cells are declared with `>>` and addressed with `::`.

**Declaration:**

```
>> cellname :: <type_mask>
```

**Assignment:**

```
cellname <- <expression>
```

**Reading a cell:**

```
::cellname
```

**Example:**

```
>> score :: #int
score <- 0
```

### 3.4 Type Masks

Types are not types — they are **masks** applied to 64-bit words.

|Mask Token           |Meaning                                  |
|---------------------|-----------------------------------------|
|`#int`               |Signed 64-bit integer                    |
|`#uint`              |Unsigned 64-bit integer                  |
|`#flt`               |IEEE 754 double                          |
|`#bit`               |Boolean (0 or 1)                         |
|`#byt`               |Raw byte                                 |
|`#txt`               |UTF-8 string (heap pointer + length)     |
|`#ptr`               |Raw memory pointer                       |
|`#void`              |No value (used in function signatures)   |
|`#any`               |Unmasked 64-bit word (dangerous, allowed)|
|`#arr[<mask>]`       |Typed array                              |
|`#map[<mask>:<mask>]`|Hash map                                 |

### 3.5 Literals

```
42          ~~ integer
42.0        ~~ float
0b1010      ~~ binary
0xFF        ~~ hex
0o17        ~~ octal
"hello"     ~~ string (UTF-8, escape with \n \t \0 \\ \" \xNN)
!!true      ~~ boolean true
!!false     ~~ boolean false
>>null      ~~ null/nil
[1, 2, 3]  ~~ array literal
{k: v}      ~~ map literal
```

### 3.6 Operators

ALL operators are symbolic. There are no word operators like `and`, `or`, `not`.

**Arithmetic:**

|Symbol|Operation     |
|------|--------------|
|`+>`  |Addition      |
|`-<`  |Subtraction   |
|`*>>` |Multiplication|
|`/<`  |Division      |
|`%^`  |Modulo        |
|`**^` |Exponentiation|

**Comparison:**

|Symbol|Operation            |
|------|---------------------|
|`==?` |Equal                |
|`!=?` |Not equal            |
|`>>?` |Greater than         |
|`<<?` |Less than            |
|`>>=?`|Greater than or equal|
|`<<=?`|Less than or equal   |

**Logical:**

|Symbol|Operation  |
|------|-----------|
|`&&^` |Logical AND|
|`||^` |Logical OR |
|`!!^` |Logical NOT|

**Bitwise:**

|Symbol|Operation  |
|------|-----------|
|`&>>` |Bitwise AND|
|`|<<` |Bitwise OR |
|`^~~` |Bitwise XOR|
|`~>>` |Bitwise NOT|
|`<<<` |Left shift |
|`>>>` |Right shift|

**Memory/Pointer:**

|Symbol|Operation     |
|------|--------------|
|`@>>` |Address-of    |
|`<<@` |Dereference   |
|`->@` |Pointer offset|

### 3.7 Blocks & Scope

Blocks are enclosed in `{|` and `|}`. There is no whitespace-based scoping.

```
{|
    ~~ code here
|}
```

Scopes do NOT inherit parent cells by default. To import a parent cell into a child scope, use the `^pull` directive:

```
{|
    ^pull score
    ~~ now ::score is accessible
|}
```

### 3.8 Control Flow

#### IF / ELSE IF / ELSE

```
?[ <condition> ]? {|
    ~~ then block
|} :?: {|
    ~~ else block
|}
```

#### CHAINED ELSE-IF

```
?[ <condition> ]? {|
    ~~...
|} :?[ <condition> ]?: {|
    ~~...
|} :?: {|
    ~~...
|}
```

#### LOOP (infinite)

```
@@loop {|
    ~~ body
|}
```

#### WHILE-STYLE LOOP

```
@@while[ <condition> ] {|
    ~~ body
|}
```

#### FOR-RANGE LOOP

```
@@range[ ::start ~> ::end ~>? ::step ] as >>i {|
    ~~ body, i is the loop cell
|}
```

(`~>` means “up to exclusive”, `~>?` means “step by”, `~>=` means “up to inclusive”)

#### FOR-EACH LOOP

```
@@each[ ::myArray ] as >>item >>idx {|
    ~~ body
|}
```

#### BREAK / CONTINUE

```
>>brk    ~~ break out of loop
>>cnt    ~~ continue to next iteration
```

#### SWITCH-LIKE: MATCH BLOCK

```
??[ ::value ] {|
    => 1 :: {| ~~ body |}
    => 2 :: {| ~~ body |}
    => "default" :: {| ~~ body |}
    =>_ :: {| ~~ wildcard/default |}
|}
```

### 3.9 Functions

Functions are called **vaults**. They do not return values — they deposit into an **output cell**.

**Declaration:**

```
vault <name>[ <param1> :: <mask>, <param2> :: <mask> ] -> <output_mask> {|
    ~~ body
    deposit <expression>   ~~ this is the "return"
|}
```

**Call:**

```
<name>![ <arg1>, <arg2> ]  ->| >>result_cell
```

The `->|` operator deposits the vault’s output into the named cell.

**Example:**

```
vault add[ a :: #int, b :: #int ] -> #int {|
    deposit ::a +> ::b
|}

>> total :: #int
add![ 3, 4 ] ->| >>total
```

**Recursive vaults** must be explicitly marked:

```
vault^rec factorial[ n :: #int ] -> #int {|
    ?[ ::n <<=? 1 ]? {|
        deposit 1
    |} :?: {|
        >> sub :: #int
        factorial![ ::n -< 1 ] ->| >>sub
        deposit ::n *>> ::sub
    |}
|}
```

**Variadic vaults:**

```
vault^va sum[ ...args :: #arr[#int] ] -> #int {|
    >> acc :: #int
    acc <- 0
    @@each[ ::args ] as >>x >>_ {|
        acc <- ::acc +> ::x
    |}
    deposit ::acc
|}
```

### 3.10 Modules & Imports

**Defining a module** (at top of file after sigil block):

```
module:: myMath
```

**Exporting a cell or vault:**

```
^export >> myCell :: #int
^export vault myFunc[ ... ] -> #int {| ... |}
```

**Importing:**

```
^use "path/to/file.m5r" as >>modAlias
```

**Accessing imported symbols:**

```
modAlias::myFunc![ args ] ->| >>result
```

### 3.11 Structs (called **Frames**)

```
frame:: Point {|
    x :: #flt
    y :: #flt
    z :: #flt
|}
```

**Instantiating a Frame:**

```
>> p :: Point
p <- Point::new![ 1.0, 2.0, 3.0 ]
```

**Accessing Frame fields:**

```
p::>x    ~~ read field x
p::>x <- 5.0   ~~ write field x
```

**Frame Vaults (methods):**

```
frame:: Point {|
    x :: #flt
    y :: #flt

    vault::self magnitude[] -> #flt {|
        deposit ( self::>x *>> self::>x ) +> ( self::>y *>> self::>y )
    |}
|}
```

Calling a frame vault:

```
>> mag :: #flt
p::>magnitude![] ->| >>mag
```

### 3.12 Error Handling

There are no exceptions. Errors are **fault cells**.

**Declaring a faultable vault:**

```
vault^fault readFile[ path :: #txt ] -> #txt {|
    ~~ ...
    ??fault "File not found: " +> ::path
    deposit ::contents
|}
```

**Handling a fault:**

```
readFile![ "data.txt" ] ->|? >>data ??catch >>err {|
    ~~ handle error stored in ::err
|}
```

**Fault propagation (bubble up):**

```
>>^fault   ~~ propagates the current fault to caller
```

### 3.13 Concurrency (called **Threads of Execution** or **ToEs**)

```
^toe {|
    ~~ this block runs concurrently
|}
```

**Synchronized cell (shared memory safe):**

```
>>^sync counter :: #int
```

**Channel (ToE communication):**

```
>>^ch myChan :: #int       ~~ declare a typed channel
myChan <<~ 42              ~~ send value into channel
myChan ~>> ::val           ~~ receive value from channel (blocks)
```

**Waiting for ToEs:**

```
^toegroup {|
    ^toe {| ... |}
    ^toe {| ... |}
|} ^join   ~~ blocks until all ToEs finish
```

### 3.14 The Execution Reversal Rule

The most alien feature of m5rcode: **the main execution block runs statements bottom-to-top by default.**

```
%%m5r
@ident: example
@ver: 1.0.0
@author: user
%%

^main {|
    print![ "world" ]    ~~ executes SECOND
    print![ "hello" ]    ~~ executes FIRST
|}
```

Output:

```
hello
world
```

To run a block **forward** (top-to-bottom), prefix it with `^fwd`:

```
^fwd ^main {|
    print![ "hello" ]   ~~ executes FIRST
    print![ "world" ]   ~~ executes SECOND
|}
```

Individual statements can be pinned to a specific execution order using **sequence stamps**:

```
[1] print![ "first" ]
[3] print![ "third" ]
[2] print![ "second" ]
```

Sequence stamps override the reversal rule and execute in stamp order.

## SECTION 5 — VIRTUAL MACHINE SPECIFICATION

### 5.1 VM Architecture

- **Stack-based** with a value stack of depth 4096
- **16 general-purpose registers**: `R0` through `R15` (64-bit each)
- **1 instruction pointer**: `IP`
- **1 stack pointer**: `SP`
- **1 frame pointer**: `FP`
- **1 fault register**: `FR` (holds current fault string pointer or 0)
- **1 flags register**: `FL` (bits: ZERO, NEG, CARRY, OVERFLOW, FAULT)
- **Heap**: Managed slab allocator, GC via tri-color mark-and-sweep
- **Call stack**: Separate from value stack, max depth 512

## SECTION 7 — BYTECODE FILE FORMAT (`.m5rc`)

Binary format, little-endian:

```
Offset  Size  Field
------  ----  -----
0x00    4     Magic: 0x4D 0x35 0x52 0x43  ("M5RC")
0x04    2     Format version (u16)
0x06    4     Flags (u32)
0x0A    4     Constant pool size (u32)
0x0E    4     Code section size (u32)
0x12    4     Symbol table size (u32)
0x16    4     Debug section size (u32)
0x1A    N     Constant pool (strings, floats, etc.)
...     N     Code section (raw bytecode)
...     N     Symbol table (name → address mappings)
...     N     Debug section (IP → source line/col mappings)
```

## SECTION 8 — GARBAGE COLLECTOR

Implement **tri-color mark-and-sweep** GC:

1. **White set**: Not yet visited (candidate for collection)
1. **Gray set**: Visited but children not yet scanned
1. **Black set**: Fully scanned, reachable

**GC Phases:**

1. **Mark roots**: all cells, registers, stack slots
1. **Scan gray**: move gray → black, push children to gray
1. **Sweep**: free all remaining white objects
1. **Compact** (optional, triggered by fragmentation threshold)

GC must run:

- Automatically when heap usage exceeds 75% of limit
- On explicit `@sys::gc![]` call
- At program termination

### 10.2 Fibonacci

```
%%m5r
@ident: fib
@ver: 1.0.0
@author: user
%%

^use "@io"

vault^rec fib[ n :: #int ] -> #int {|
    ?[ ::n <<=? 1 ]? {|
        deposit ::n
    |} :?: {|
        >> a :: #int
        >> b :: #int
        fib![ ::n -< 1 ] ->| >>a
        fib![ ::n -< 2 ] ->| >>b
        deposit ::a +> ::b
    |}
|}

^fwd ^main {|
    >> i :: #int
    >> result :: #int
    @@range[ 0 ~> 10 ~>? 1 ] as >>i {|
        fib![ ::i ] ->| >>result
        @io::println![ ::result ]
    |}
|}
```

### 10.3 HTTP Server (skeleton)

```
%%m5r
@ident: webserver
@ver: 1.0.0
@author: user
%%

^use "@net"
^use "@io"
^use "@str"

vault handleReq[ req :: #any ] -> #txt {|
    deposit "HTTP/1.1 200 OK\r\nContent-Length: 2\r\n\r\nOK"
|}

^fwd ^main {|
    >> srv :: #any
    @net::tcpListen![ 8080, handleReq ] ->|? >>srv ??catch >>err {|
        @io::eprintln![ ::err ]
        @sys::exit![ 1 ]
    |}
    @io::println![ "m5rcode server on :8080" ]
|}
```

## SECTION 11 — ERROR MESSAGES

All errors must follow this exact format:

```
[M5R-FAULT] <ERROR_CODE>: <message>
  --> <filename>.m5r:<line>:<col>
   | <source_line>
   |  ^~~ <indicator>
```

Error codes:

- `E001` — Unexpected token
- `E002` — Undeclared cell
- `E003` — Type mask mismatch
- `E004` — Vault not found
- `E005` — Wrong argument count
- `E006` — Missing `deposit` in vault
- `E007` — Missing file sigil block
- `E008` — Division by zero (runtime)
- `E009` — Stack overflow (runtime)
- `E010` — Null dereference (runtime)
- `E011` — Array index out of bounds (runtime)
- `E012` — Fault not caught
- `E013` — Cell used before declaration
- `E014` — `^pull` used for non-existent parent cell
- `E015` — Import not found
- `E016` — Circular import detected
- `E017` — Invalid type mask
- `E018` — Unmatched block `{|` / `|}`
- `E019` — ToE channel type mismatch
- `E020` — Bytecode version mismatch
