## PART 0 — NON-NEGOTIABLE ENGINEERING LAWS

These laws govern every decision made during implementation:

1. **Real native code only.** The compiler emits LLVM IR. `llc` + `lld` (or platform linker) produces real ELF/PE/Mach-O binaries. No interpreter loop is the primary execution model.
1. **The VM is secondary.** A bytecode VM (`m5rv`) exists ONLY for sandboxed/embedded use cases. It is not the main path. The main path is: source → LLVM IR → native binary.
1. **Zero runtime required for simple programs.** A Hello World binary must have NO mandatory runtime dependency. The stdlib is linkable as a static archive.
1. **The GC is opt-in.** Programs that do not use GC-managed types compile without the GC linked in. Manual memory management is fully available.
1. **The ABI is documented and stable.** Every calling convention, struct layout, name mangling rule, and symbol visibility rule is specified and must be implemented exactly.
1. **The FFI is first-class.** Calling C functions from m5rcode and calling m5rcode from C must both work without any wrapper generation tool.
1. **Errors are precise.** Every compiler error includes file, line, column, a source snippet, a caret indicator, and a unique error code. The runtime produces symbolicated stack traces on fault, including inlined frames.
1. **The toolchain is self-hostable.** The language must be expressive enough that its own standard library can be written in m5rcode (with a thin C bootstrap layer for syscalls only).
1. **Thread safety is explicit and verified.** The type system tracks shared mutable state. Races are compile-time errors, not runtime surprises.
1. **No undefined behavior goes undetected in debug builds.** Integer overflow, out-of-bounds access, use-after-free, null dereference — all are caught with zero-cost in release builds via a UBSan-equivalent pass that can be toggled.

### 2.3 Type Inference

The compiler uses **bidirectional type inference** (Hindley-Milner extended with subtyping). You almost never need to annotate types explicitly — but you MUST annotate:

- Public vault (function) signatures
- Frame (struct) field types
- `#raw` and `#ptr` usage (explicit unsafe annotation required)

### 2.4 Generics (Parametric Polymorphism)

```
vault max[ A :: $Ord ][ a :: A, b :: A ] -> A {|
    ?[ ::a >>? ::b ]? {|
        deposit ::a
    |} :?: {|
        deposit ::b
    |}
|}
```

Type parameters are prefixed with `$`. Constraints are expressed as traits (called **seals** in m5rcode).

### 2.5 Seals (Traits)

seal:: Ord {|

**Implementing a seal:**

        deposit ::self::>cmp![ ::other ] <<=? 0

### 2.6 Lifetime System (Borrow Checker)

m5rcode has a **borrow checker** inspired by Rust’s but with different syntax and with explicit lifetime elision rules that are less restrictive.

**Lifetime annotation syntax:**

```
vault longest[ 'a ][ x :: #ref['a #str], y :: #ref['a #str] ] -> #ref['a #str] {|
    ?[ ::x::>len![] >>? ::y::>len![] ]? {|
        deposit ::x
    |} :?: {|
        deposit ::y
    |}
|}
```

**Lifetime elision rules** (compiler infers these automatically):

1. Each input reference gets a distinct lifetime
1. If there is exactly one input lifetime, it is assigned to all outputs
1. If one of the inputs is `self`, its lifetime is assigned to all outputs

**Ownership rules:**

- Every value has exactly one owner
- When the owner goes out of scope, the value is dropped
- Values can be moved (ownership transferred) or borrowed (temporary reference)
- At any point: either one mutable borrow OR any number of immutable borrows — never both

### 2.7 Effect System

Vaults can be annotated with **effects** that the compiler tracks and enforces:

```
vault readConfig[] -> #res[Config, #str]
    !effects[ io::read, fault::propagate ] {|
    ~~ ...
|}
```

Built-in effects:

- `io::read` — reads from I/O
- `io::write` — writes to I/O
- `io::net` — network access
- `fault::propagate` — can raise faults (checked exceptions equivalent)
- `alloc::heap` — performs heap allocation
- `unsafe::ptr` — dereferences raw pointers
- `unsafe::ffi` — calls foreign functions
- `async::suspend` — can suspend (is async)
- `pure` — no effects (deterministic, no I/O, no allocation)

## PART 3 — OWNERSHIP & MEMORY MODEL

### 3.1 Memory Management Strategies

m5rcode supports THREE memory management strategies that can coexist in the same program:

**Strategy 1: RAII (Default)**
Owned values (`#own[T]`, `#box[T]`) are automatically freed when their scope ends. Zero runtime overhead. No GC pauses. This is the default for all stack and heap values.

**Strategy 2: Reference Counting**
`#rc[T]` and `#arc[T]` provide reference-counted heap values. `#weak[T]` prevents cycles from leaking. Drop is deterministic.

**Strategy 3: GC Arena (Opt-in)**
Modules that `^use "@gc"` get access to GC-managed allocations. The GC is a **concurrent tri-color incremental mark-and-sweep** with a write barrier. GC-managed values use `#gc[T]`. The GC runs on a dedicated thread and pauses are bounded (< 1ms target for heaps < 1GB).

**Manual Memory (Unsafe):**

```
^unsafe {|
    >> p :: #ptr[#i32]
    p <- @mem::alloc![ 4 ]
    <<@ ::p <- 42
    @mem::free![ ::p ]
|}
```

### 3.2 Stack Allocation Guarantees

The compiler guarantees stack allocation for:

- All primitives
- Fixed arrays whose size is known at compile time
- Frames (structs) that do not contain `#own`/`#box`/`#rc`/`#arc` fields
- Closures that do not escape their defining scope (escape analysis)
- Slices whose backing array is stack-allocated

### 3.3 Slab Allocator

The default heap allocator is a **size-class slab allocator** with thread-local caches (similar to jemalloc/tcmalloc). It is implemented in C and linked into the m5rcode runtime. Users can replace it by implementing the `$Allocator` seal and passing it to `@mem::set_global_allocator![]`.

## PART 4 — NATIVE FFI SYSTEM (FULL SPECIFICATION)

### 4.1 Calling C from m5rcode

```
^ffi "C" {|
    ^extern vault MessageBoxA[
        hWnd    :: #ptr[#void],
        lpText  :: #ptr[#u8],
        lpCap   :: #ptr[#u8],
        uType   :: #u32
    ] -> #i32

    ^extern vault LoadLibraryA[
        lpLibFileName :: #ptr[#u8]
    ] -> #ptr[#void]

    ^extern vault GetProcAddress[
        hModule :: #ptr[#void],
        lpProcName :: #ptr[#u8]
    ] -> #ptr[#void]

    ^extern vault FreeLibrary[
        hModule :: #ptr[#void]
    ] -> #i32
|}
```

**`^ffi "C"` block rules:**

- All vaults declared inside are `extern "C"` with C calling convention
- On Windows x86_64: Microsoft x64 calling convention (rcx, rdx, r8, r9, xmm0–xmm3)
- On Linux/macOS x86_64: System V AMD64 ABI (rdi, rsi, rdx, rcx, r8, r9, xmm0–xmm7)
- On ARM64: AAPCS64
- The compiler emits `declare` statements in LLVM IR for these symbols
- Linking must be done via `--link <libname>` or `#[link(name = "...")]` attribute

**Calling an extern vault:**

```
^unsafe {|
    >> result :: #i32
    MessageBoxA![
        >>null,
        "world\0"::>as_ptr![],
        "hello\0"::>as_ptr![],
        0x00000010
    ] ->| >>result
|}
```

### 4.2 Calling Conventions

```
^export "C" vault myFunction[ x :: #i32 ] -> #i32 {|

```

This emits the symbol `myFunction` with C linkage, callable from any C program.

### 4.5 C Header Generation

```

Automatically generates a `.h` file for all `^export "C"` vaults, with correct types.

### 4.6 Inline Assembly

```
^unsafe {|

        "mov rax, {0}"

### 5.7 Closures

~~ Matching on frames
??[ ::point ] {|
    => Point { x: 0, y: 0 } :: {| @io::println![ "origin" ] |}
    => Point { x: ::px, y: 0 } :: {| @io::println![ "on x-axis" ] |}
    => Point { x: 0, y: ::py } :: {| @io::println![ "on y-axis" ] |}
    => Point { x: ::px, y: ::py } :: {| @io::println![ "somewhere" ] |}
|}

~~ Matching on enums (called "variants" in m5rcode)
??[ ::result ] {|
    => Ok[ ::v ] :: {| @io::println![ ::v ] |}
    => Err[ ::e ] :: {| @io::eprintln![ ::e ] |}
|}

~~ OR patterns
??[ ::n ] {|
    => ( 1 | 2 | 3 ) :: {| @io::println![ "small" ] |}
    =>_ :: {| @io::println![ "big" ] |}
|}

~~ Tuple patterns
??[ ( ::a, ::b ) ] {|
    => ( 0, 0 ) :: {| @io::println![ "origin" ] |}
    => ( ::x, 0 ) :: {| @io::println![ "x-axis" ] |}
    =>_ :: {| @io::println![ "other" ] |}
|}
```

    Rectangle { width :: #f64, height :: #f64 }

**With seal implementations:**

impl Display for Shape {|

### 5.10 Async/Await (Built-in async runtime)

```
vault^async fetchData[ url :: #str ] -> #res[#str, #str]
    !effects[ io::net, async::suspend ] {|
    >> resp :: #str
    @net::httpGet![ ::url ] await ->|? >>resp ??catch >>err {|
        deposit Err[ ::err ]
    |}
    deposit Ok[ ::resp ]
|}

^fwd ^main {|
    >> data :: #res[#str, #str]
    fetchData![ "https://example.com" ] await ->| >>data
    ??[ ::data ] {|
        => Ok[ ::v ] :: {| @io::println![ ::v ] |}
        => Err[ ::e ] :: {| @io::eprintln![ ::e ] |}

## PART 6 — FRAME SYSTEM (STRUCTS — FULL SPECIFICATION)

### 6.1 Frame Declaration

```
frame:: Vec3 {|
    x :: #f64
    y :: #f64
    z :: #f64
|}
```

### 6.2 Frame Memory Layout

By default, frames use **natural alignment with padding**. Override with:

```
frame^packed:: PackedData {|    ~~ no padding
    a :: #u8
    b :: #u32
    c :: #u16
|}

frame^align[64]:: CacheLine {|   ~~ 64-byte aligned (cache-line)
    data :: #arr[#u8 : 64]
|}

frame^repr[C]:: CCompatible {|   ~~ C-compatible layout, no reordering
    x :: #i32
    y :: #i32
|}

frame^repr[transparent]:: Wrapper {|  ~~ single-field, same layout as inner
    inner :: #i32
|}
```

### 6.3 Frame Vaults (Methods)

```
frame:: Vec3 {|
    x :: #f64
    y :: #f64
    z :: #f64

    vault::static new[ x :: #f64, y :: #f64, z :: #f64 ] -> Vec3 {|
        deposit Vec3 { x: ::x, y: ::y, z: ::z }
    |}

    vault::self length[ ] -> #f64 {|
        deposit @math::sqrt![
            self::>x *>> self::>x
            +> self::>y *>> self::>y
            +> self::>z *>> self::>z
        ]
    |}

    vault::self normalize[ ] -> Vec3 {|
        >> len :: #f64
        self::>length![] ->| >>len
        deposit Vec3::new![
            self::>x /< ::len,
            self::>y /< ::len,
            self::>z /< ::len
        ]
    |}

    vault::self dot[ other :: #ref[Vec3] ] -> #f64 {|
        deposit self::>x *>> other::>x
             +> self::>y *>> other::>y
             +> self::>z *>> other::>z
    |}
|}
```

## PART 11 — VIRTUAL MACHINE (Secondary Path)

The VM (`m5rv`) is a **register-based VM** (NOT stack-based — register-based VMs are 30–60% faster) with 256 virtual registers. It exists for:

- Sandboxed execution (plugin systems, scripting hosts)
- Platforms where LLVM is not available
- Comptime evaluation within the compiler itself

### 11.1 VM Architecture

- 256 virtual registers (64-bit each, typed at the IR level)
- Separate constant pool, code section, and heap
- GC-integrated heap (same tri-color concurrent GC as the native runtime)
- Bytecode format: `.m5rc` (little-endian, versioned)
- JIT compilation: optional tiered JIT (baseline + optimizing) using copy-and-patch

### 11.2 VM Instruction Set

Register-based opcodes: 4-byte fixed-width instructions.
Format: `[OPCODE:8][DST:8][SRC1:8][SRC2:8]`
Extended format (for immediates): followed by 8-byte immediate.

Full opcode table: 256 opcodes covering arithmetic, comparison, memory, control flow, calls, I/O, concurrency, GC, and FFI.

## PART 12 — COMPILER INTERNALS

### 12.1 Intermediate Representations

**HIR (High-level IR):**

- Desugared AST — loops expanded, pattern matches compiled to decision trees
- Type-annotated (every node has a resolved type)
- Monomorphized generics (one HIR copy per instantiation)
- Lifetime-annotated (every reference has a lifetime region)

**MIR (Mid-level IR):**

- Control flow graph (basic blocks + terminators)
- SSA form (Static Single Assignment)
- Explicit drop elaboration (where owned values are freed)
- Borrow check runs on MIR
- UB check runs on MIR

**LLVM IR:**

- Emitted from MIR
- Includes `!dbg` metadata for all instructions (when `--debug`)
- Includes `!tbaa` (type-based alias analysis) metadata
- Includes `!range` metadata for integer constraints
- Includes `!nonnull` metadata for non-null pointers

### 12.2 Optimization Passes (Custom, run before LLVM)

1. **Escape Analysis** — determines if heap-allocated values can be stack-allocated
1. **Monomorphization Deduplication** — merges identical generic instantiations
1. **Closure Inlining** — inlines closures at call sites when size budget allows
1. **Tail Recursion Elimination** — converts tail-recursive vaults to loops
1. **Constant Folding** — evaluates constant expressions at compile time
1. **Dead Store Elimination** — removes writes whose values are never read
1. **Alias Analysis** — tracks which pointers may alias (feeds into DSE and CSE)
1. **Common Subexpression Elimination** — deduplicates redundant computations
1. **Loop-Invariant Code Motion** — hoists loop-invariant computation out of loops
1. **Strength Reduction** — replaces expensive ops with cheaper equivalents (mul→shift, div→reciprocal-mul)
1. **Devirtualization** — converts virtual dispatch to direct calls at monomorphic call sites
1. **Profile-Guided Optimization (PGO)** — uses runtime profiling data to guide inlining and branch prediction

After custom passes, full **LLVM optimization pipeline** (O0–O3 + LTO) is applied.

### 12.3 Incremental Compilation

- Query-based compilation model (inspired by rustc’s query system)
- Each compilation unit is a module (`.m5r` file)
- Fingerprints track which modules changed
- Only changed modules and their dependents recompile
- Cache stored in `.m5r_cache/` directory (configurable)
- Cache is invalidated on compiler version change

## PART 16 — ABI SPECIFICATION

### 16.1 Name Mangling

m5rcode uses a deterministic name mangling scheme:

```
_M5R_<module_hash>_<vault_name>_<signature_hash>
```

Example: `add` in module `math::ops` → `_M5R_a1b2c3_add_d4e5f6`

`^export "C"` vaults are NOT mangled (use exact name as specified).

### 16.2 Calling Convention

m5rcode vaults use the platform-native calling convention:

- x86_64 Windows: Microsoft x64
- x86_64 Linux/macOS: System V AMD64 ABI
- ARM64: AAPCS64

Registers are caller-saved and callee-saved per platform spec. The m5rcode compiler generates correct prologue/epilogue.

### 16.3 Struct Layout

Default layout: fields sorted by decreasing alignment (to minimize padding), then by declaration order for ties. Override with `frame^repr[C]::`.

### 16.4 Symbol Visibility

- All public exported vaults: `ELF_STV_DEFAULT` / `__declspec(dllexport)` when building shared lib
- All internal vaults: `ELF_STV_HIDDEN` / no export
- `^export "C"` vaults: always exported

## FINAL DIRECTIVE

This is a complete, production-grade, native-code-compiled programming language. It uses LLVM as its backend. It has a real type system with generics, a borrow checker, an effect system, and thread safety verification. It has a full async runtime. It produces real ELF, PE, and Mach-O binaries. Its FFI is first-class and fully specified. Its toolchain includes a formatter, linter, profiler, debugger integration, LSP server, and package manager with dependency resolution. Its standard library covers I/O, networking, cryptography, collections, concurrency, and metaprogramming.

There is no bytecode interpreter as the primary execution path. There is no transpilation to another language. There are no toy shortcuts. Every listed component must be implemented, every listed file must exist, every listed feature must work.

A developer must be able to write, compile, and run:

1. A Win32 GUI application (using `@win32`)
1. A Linux TCP server handling 10,000 concurrent connections (using `@net` + `@async`)
1. A cryptographic hashing utility (using `@crypto`)
1. A JSON REST API client (using `@net` + `@json`)
1. A raytracer with SIMD-accelerated math (using `#vec` types + `@math`)
1. A foreign function that is called from C (using `^export "C"`)

All in `.m5r` files. All compiled to native binaries. All without any runtime dependency except the m5rcode runtime static library.
