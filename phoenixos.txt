Create a fully custom operating system project from scratch that is inspired by the very early spirit of Linux as “mostly a kernel first” system, but is NOT a Linux distribution. It must be its own standalone operating system project with its own identity, source tree, boot flow, userspace, init system, desktop environment, utilities, build tooling, branding, and documentation. It may be based on the general Linux kernel architecture and use a heavily modified custom Linux kernel codebase as the hardware abstraction and core system base, but the final result must feel like its own original operating system rather than “Linux + packages”. Do not make it a distro, do not base it on Ubuntu, Debian, Arch, Fedora, Alpine, or any existing distro layout. Build a unique OS.

Project language stack:
- C for low-level kernel components, core runtime, drivers, and system libraries
- Rust for safer modern subsystems, selected services, IPC helpers, package tooling, desktop shell logic, and selected userspace components
- x86 Assembly for boot path, early startup, context switching stubs, interrupt/exception entry code, and other critical low-level routines

Primary design goals:
1. A real custom operating system identity
2. Early-Linux-like simplicity in spirit, especially in kernel structure and philosophy
3. A custom desktop environment inspired visually and behaviorally by Windows 1.0 and Windows 3.0
4. Small, understandable, hackable codebase
5. Bootable on modern x86_64 PCs and in VMs like QEMU, VirtualBox, and VMware
6. Clean architecture and educational value
7. Strong separation between kernel, core services, and GUI stack
8. Buildable by a single developer or small team
9. No dependency on GNU/Linux userspace
10. Should feel like an alternate-history retro desktop OS with modern code discipline

High-level concept:
Design an operating system that uses a custom-maintained Linux-derived kernel core as its base, but everything around it must be custom: boot experience, init, service manager, shell, libraries where practical, system utilities, configuration format, window manager, desktop environment, file manager, control panel, installer image, package format, branding, and developer tooling. Think “what if early Linux grew into a unique retro-GUI operating system instead of powering modern distributions”.

Target platform:
- x86_64 first-class support
- Optional i686 compatibility layer or secondary target if feasible
- BIOS and UEFI boot support if possible, otherwise choose one primary path first and clearly document roadmap for the other
- VM-first development target, with real hardware support as a second stage
- PS/2 and USB keyboard/mouse support
- VGA/framebuffer first, with optional VESA/simple graphics layer
- SATA/AHCI and basic virtual disk drivers
- EXT2-like or custom simple filesystem support for early versions

Kernel requirements:
The kernel must be described as a custom Linux-kernel-based core that is heavily modified and stripped down for this OS. Keep the design simple and educational. Include:
- boot and early init
- memory management
- scheduler
- interrupt handling
- syscall interface
- timer support
- process and thread model
- VFS layer
- minimal driver model
- TTY/console subsystem
- framebuffer or graphics bridge for GUI
- IPC primitives
- module support optional, monolithic kernel acceptable
- clear explanation of what parts are kernel-space and what parts are user-space

Kernel philosophy:
- Keep it small and understandable
- Prefer clarity over enterprise complexity
- Avoid unnecessary modern Linux ecosystem baggage
- Focus on direct control and low dependency count
- Preserve a “first principles OS hacker” feel
